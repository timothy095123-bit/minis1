Topic: Python Inheritance Example
Here is a Python class using inheritance and polymorphism:

class Animal:
    def __init__(self, name):
        self.name = name
    
    def speak(self):
        raise NotImplementedError("Subclass must implement")

class Dog(Animal):
    def speak(self):
        return f"{self.name} says Woof!"

class Cat(Animal):
    def speak(self):
        return f"{self.name} says Meow!"

This example shows how child classes inherit from a parent class and override methods to provide specific behavior.

Topic: Python Decorator Pattern
Here is a Python decorator for memoization:

def memoize(func):
    cache = {}
    def wrapper(*args):
        if args not in cache:
            cache[args] = func(*args)
        return cache[args]
    return wrapper

@memoize
def fibonacci(n):
    if n < 2:
        return n
    return fibonacci(n-1) + fibonacci(n-2)

Decorators wrap functions to add extra functionality without modifying the original function.

Topic: Python Context Manager

class DatabaseConnection:

Topic: JavaScript Async Example
Here is a JavaScript class with async/await:

class UserService {
    constructor(apiUrl) {
        this.apiUrl = apiUrl;
    }

        }
    }
}

Topic: Java Strategy Pattern
Here is a Java design pattern using Strategy Pattern:

interface PaymentStrategy {
    void pay(int amount);
}

class CreditCardPayment implements PaymentStrategy {
    private String cardNumber;
    
    public CreditCardPayment(String cardNumber) {
        this.cardNumber = cardNumber;
    }
    
    @Override
    public void pay(int amount) {
        System.out.println("Paid " + amount + " using Credit Card");
    }
}

class PaymentProcessor {
    private PaymentStrategy strategy;
    
    public void setStrategy(PaymentStrategy strategy) {
        this.strategy = strategy;
    }
    
    public void processPayment(int amount) {
        strategy.pay(amount);
    }
}

The Strategy Pattern allows you to change algorithms at runtime.

Topic: C# LINQ Extensions
Here is a C# LINQ and extension method example:

public static class EnumerableExtensions {
    public static IEnumerable<T> WhereNotNull<T>(this IEnumerable<T> source) {
        return source.Where(item => item != null);
    }
}

public class DataProcessor {
    public List<User> GetActiveUsers(IEnumerable<User> users) {
        return users
            .WhereNotNull()
            .Where(u => u.IsActive)
            .OrderBy(u => u.LastName)
            .ThenBy(u => u.FirstName)
            .ToList();
    }
}

Topic: Dependency Injection
Question: Why should we use dependency injection instead of creating objects directly?
Answer: Dependency injection improves testability because we can mock dependencies. It reduces coupling between classes, making code more maintainable. It follows the Dependency Inversion Principle from SOLID. When requirements change, we only modify the injection configuration, not the class itself.

Topic: Composition vs Inheritance
Question: What is the difference between composition and inheritance?
Answer: Composition means "has-a" relationship while inheritance means "is-a" relationship. Composition is more flexible because you can change behavior at runtime. Inheritance creates tight coupling between parent and child classes. The rule is favor composition over inheritance for better flexibility and maintainability.

Topic: Async Benefits
Question: Why use async/await instead of callbacks?
Answer: Async/await makes asynchronous code look synchronous, improving readability. It avoids callback hell and makes error handling easier with try/catch. The code flow is more linear and easier to debug.

Topic: Database Selection
Question: When should you use a NoSQL database instead of SQL?
Answer: Use NoSQL when you need horizontal scalability, flexible schema, or handle unstructured data. SQL is better when you need ACID transactions, complex joins, or strict data consistency. Consider your access patterns and scale requirements before choosing.

Topic: Pure Functions
Question: What makes a function pure in functional programming?

Problem: Your API returns 500 errors intermittently.
Solution: Check logs for error messages and stack traces. Identify if it's database connection timeout or memory issues. Monitor resource usage like CPU, memory, and connections. Add retry logic for transient failures. Implement circuit breaker pattern for failing dependencies. Use correlation IDs to trace requests across services.

Solution: Analyze slow query logs to identify bottlenecks. Add indexes on frequently queried columns. Use EXPLAIN to understand query execution plan. Avoid N+1 queries by using joins or eager loading. Implement caching for frequently accessed data. Consider denormalization for read-heavy workloads. Use connection pooling to reduce overhead.

Topic: Mutable Default Arguments
Problem: Why is this code problematic?

def add_item(item, items=[]):
    items.append(item)
    return items

Explanation: Default arguments are evaluated once at function definition, not each call. The list is shared across all calls, causing unexpected behavior.

Solution:

def add_item(item, items=None):
    if items is None:
        items = []
    items.append(item)
    return items

Topic: Input Validation
Question: Why validate input on both client and server?
Answer: Client-side validation provides immediate feedback and better UX. However, it can be bypassed, so server-side validation is mandatory for security. Always validate, sanitize, and escape user input on the server. Defense in depth means validate everywhere, but trust nothing from the client.

Topic: Performance Optimization
Problem: Why is this loop slow?

result = []
for item in large_list:
    if item not in result:
        result.append(item)

Explanation: The in operator on a list performs linear search O(n). Inside a loop of n items, this becomes O(n squared). Using a set provides O(1) lookup, reducing total complexity to O(n).

Optimized Solution:

result = list(set(large_list))

Or preserve order:

result = list(dict.fromkeys(large_list))

Topic: Event-Driven Architecture
Question: When to use event-driven architecture?
Answer: Use events when you need loose coupling between services, async processing, or audit trails. Events enable eventual consistency and scalability. However, they add complexity in debugging and ordering guarantees. Best for systems with clear domain events.

Topic: Sorting Algorithms
Question: How to choose sorting algorithm?
Analysis: Quick Sort has O(n log n) average time and is good for general purpose. Merge Sort has O(n log n) worst case, is stable, but needs extra space. Heap Sort has O(n log n) worst case and is in-place but not stable. Tim Sort is adaptive and best for partially sorted data. Counting Sort has O(n+k) for integers with limited range. Consider data size, memory constraints, and stability requirements.

Topic: Exception Handling
Question: Try/catch vs validation?
Answer: Use validation for expected scenarios like user input. Use exceptions for unexpected errors like network failures. Exceptions have performance overhead, so do not use them for control flow. Validate early, fail fast, and provide meaningful error messages.

Topic: Thread Safety
Problem: Race condition in counter increment

counter = 0
def increment():
    global counter
    counter += 1

Explanation: The += operation is not atomic. Multiple threads can read the same value before incrementing, causing lost updates.

Solution with Lock:

from threading import Lock

counter = 0
lock = Lock()

def increment():
    global counter
    with lock:
        counter += 1

A lock ensures mutual exclusion, making the operation atomic.

Topic: Factory vs Builder Pattern
Question: When to use Factory vs Builder?
Answer: Factory creates objects in one step, good for simple object creation with different types. Builder constructs complex objects step-by-step, good for objects with many optional parameters. Builder provides fluent interface and makes construction explicit. Choose based on object complexity.

Topic: Memory Leaks
Problem: Why does this JavaScript code leak memory?

function createHandler() {
    const bigArray = new Array(1000000);
    return function() {
        console.log(bigArray[0]);
    }
}

Explanation: The returned function forms a closure that captures bigArray. Even if bigArray is not needed, it remains in memory. Solution is to only capture what you need or set references to null when done.

Topic: HTTP Methods
Question: Why use POST instead of GET for mutations?
Answer: GET should be idempotent and safe with no side effects. POST indicates state change and prevents accidental mutations from browser prefetch or caching. PUT and DELETE are also idempotent for updates and deletions. Follow HTTP semantics for predictable API behavior.

Topic: Code Smells
Problem: God Object Antipattern
Explanation: A class that knows too much or does too many things violates Single Responsibility Principle. It becomes hard to test, maintain, and understand.
Solution: Break into smaller classes with focused responsibilities. Use composition to combine behaviors. Apply SOLID principles for better design.

Topic: Testing Strategy
Question: Unit vs Integration vs E2E tests?
Answer: Unit tests are fast, isolated, and test individual functions. Integration tests verify components work together. E2E tests validate entire user flows but are slow and brittle. Use testing pyramid with many unit tests, some integration tests, and few E2E tests. Balance coverage with execution speed.

Topic: Caching Strategy
Question: When and where to cache?
Analysis: Browser cache is for static assets and CDN. Application cache uses Redis or Memcached for database queries. Database cache stores query results. CDN cache distributes content geographically. Consider cache invalidation strategy, TTL, and consistency requirements. Cache invalidation is one of the hardest problems in computer science.

Topic: Code Complexity
Concept: Cyclomatic complexity measures number of independent paths through code. High complexity indicates code is hard to test and maintain. Reduce by extracting functions, using guard clauses, and simplifying conditionals. Aim for complexity under 10 per function.

Topic: Refactoring vs Rewrite
Question: When to refactor vs rewrite?
Answer: Refactor when code is mostly good but needs improvement. Rewrite when cost of maintaining exceeds rebuild cost. Consider business continuity, team knowledge, and technical debt. Strangler pattern allows gradual replacement. Never do big-bang rewrites without incremental delivery.
